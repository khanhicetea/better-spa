# Repositories

Generic, type-safe access layer over Kysely. Repositories are the default data path for RPC handlers — reach for raw Kysely only when there is no equivalent method here.

## Conventions

- Register every domain in `index.ts` so it shows up on `context.repos`.
- Use the generic `Repository` (`repository.ts`) unless the domain needs reusable complex queries that justify a subclass.
- Schema types are handwritten in `src/server/db/schema/`. Do not run Kysely codegen.
- Identifiers are generated by callers (typically `generateUUID()`), not by the repository.
- All public methods accept `camelCase` keys; the `CamelCasePlugin` translates to `snake_case` columns at the SQL boundary.

## Common Method Map

| Need                 | Method                        |
| -------------------- | ----------------------------- |
| One row by id        | `findById`, `findByIdOrFail`  |
| One row by predicate | `findOne`, `findOneOrFail`    |
| Many rows            | `find`, `findAll`             |
| Specific columns     | `findSelect`                  |
| Pagination           | `findPaginated`               |
| Count / existence    | `count`, `exists`, `existsBy` |
| Insert               | `insertReturn`, `insertMany`  |
| Update               | `updateById`, `updateMany`    |
| Delete               | `deleteById`, `deleteMany`    |
| Insert-or-update     | `upsert`                      |

`where` accepts either a partial row (equality match across keys) or a query-builder callback for anything more complex (joins, `in`, `like`, ordering composed earlier in the chain).

## Cross-Repo Access

Inside a custom repository, use `this.repos.<other>` to reach a sibling. The factory in `index.ts` injects the reference after construction, so there is no circular import.
